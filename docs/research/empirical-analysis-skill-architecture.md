# 实证分析 Skill 架构调研草稿

## 背景

本文档记录 Starlane Skills 第一版调研结果。

项目目标是做一组面向实证分析流程的开源 Agent Skills，让 Codex、Claude Code、Cursor 这类 Agent 工具能够辅助用户完成可复现的实证研究任务。

第一版优先服务经管类实证研究场景，尤其是本科毕业论文、课程论文中常见的回归分析任务。

第一版的典型场景是：

> 用户已经有一份数据和大致研究题目。用户提供变量角色信息后，Agent 根据结构化脚本和工作流完成实证分析，枚举模型组合，汇总最佳结果，并产出可复现的代码、表格和结果解释。

这个项目不是普通聊天助手。核心能力应该是：

- 显式变量映射
- 可执行脚本
- 模型组合枚举
- 结果筛选策略
- 可解释交付物
- 可复现分析流程

## 目标用户

第一版目标用户是：

- 使用 Codex、Claude Code、Cursor 或类似 Agent 工具的人
- 本地有 Stata 或兼容的 Stata 执行环境
- 正在做经管类实证分析
- 已经有数据，或能提供待分析数据
- 至少能提供基础变量角色信息

初始用户以本科生为主，尤其是需要完成毕业论文或课程论文实证部分的经管类学生。后续可以扩展到研究生、博士生、导师和研究团队。

## 第一版范围

第一版先覆盖“用户已有数据 + 已有大概研究方向”的场景。

范围内：

- 澄清研究目标
- 收集数据和变量角色信息
- 校验必需变量映射
- 检查数据质量和面板结构
- 执行回归枚举脚本
- 汇总候选模型组合
- 选择并解释推荐结果
- 生成可复现源代码和结果表
- 生成适合实证章节写作的结果解释

第一版不覆盖：

- 从零选题
- 寻找或购买数据
- 编造数据
- 保证结果显著
- 保证论文通过
- 从头到尾代写整篇论文
- 绕过学校、课程或机构规范

## 完整实证分析流程

在拆分 skills 之前，需要先定义完整链路。

### 1. 需求澄清

Agent 需要确认：

- 研究主题
- 论文类型
- 用户目标
- 预期关系方向
- 已有数据情况
- 是否已有指定方法
- 是否有课程、学校或导师要求

这一步的目标是判断用户能否直接进入执行阶段，还是需要先补研究设计。

### 2. 数据接收

用户需要提供：

- 数据文件
- 变量说明
- 数据来源
- 样本范围
- 面板结构
- 已知限制

第一版可支持的数据格式取决于执行脚本。当前调研链路中，数据格式可优先覆盖 `.dta`、`.xlsx`、`.xls`、`.csv`。

### 3. 变量映射

实证执行依赖明确的变量角色。

必填字段：

- `y`：因变量
- `x`：核心自变量
- `cv`：控制变量
- `panelvar`：个体变量
- `timevar`：时间变量

推荐字段：

- `coef_direction`：预期系数方向，例如 `positive` 或 `negative`

可选字段：

- `cv_fixed`：固定必选控制变量
- `cv_min_count`：最少控制变量数量
- `meds`：中介变量
- `mods`：调节变量
- `heterogeneity_discrete`：离散分组异质性变量
- `heterogeneity_discrete_values`：异质性分组取值
- `alt_y`：替代因变量
- `alt_x`：替代核心自变量
- `rob_vars`：稳健性变量
- `y_ln`：是否做因变量取对数稳健性
- `x_ln`：是否做核心自变量取对数稳健性
- `rob_year_range`：稳健性样本期
- `iv`：工具变量

第一版不应该要求用户理解所有计量细节。用户需要提供变量角色，Agent 负责校验、提醒和追问。

### 4. 数据体检

执行前，Agent 应检查：

- 映射变量是否存在
- 变量类型是否合适
- 缺失值情况
- 样本量
- 面板个体-时间是否重复
- 个体和年份覆盖情况
- 是否适合取对数
- 控制变量数量是否合理
- 可选变量是否适合对应检验

第一版不单独把数据体检做成独立产品。它是回归执行前的必要步骤。

### 5. 研究设计

流程应支持：

- 基准模型设定
- 固定效应
- 聚类标准误或稳健标准误
- 控制变量策略
- 中介分析
- 调节分析
- 异质性分析
- 稳健性检验
- 在用户提供工具变量时做内生性检验

第一版不能过度承诺因果识别。只有在识别策略充分时，才可以使用因果表述。

### 6. 两阶段执行

核心执行层通过两阶段 workflow 交付结果。

当前 workflow 支持：

- 加载数据
- 校验必需参数
- 确认执行后端：Python 或 Stata
- 枚举控制变量子集
- 枚举 VCE 类型
- 对弱基准组合进行剪枝
- 运行基准回归和可选扩展分析
- 对结果组合打分
- 输出透明的 `combination_summary.csv`
- 等待用户选择候选行
- 根据 `cv_idx` 和 `vce_idx` 生成最终结果
- 输出最终结果和可复现源码

Stata 后端当前可以生成可读 `.do` 文件，并在本地 Stata 环境支持时导出 Word 回归表。Python 后端应遵守同一套 summary/final 语义。

关键产品判断：

> 系统不是只靠自然语言推理，而是把用户提供的变量映射转换为脚本参数，再通过确定性执行流程产出结果。

### 7. 结果选择

Agent 不能无脑选择最高分结果。

结果选择应综合考虑：

- 得分排序
- 系数方向
- 显著性水平
- 稳健性一致性
- 控制变量数量和合理性
- VCE 类型
- 样本量稳定性
- 结果是否过拟合或过于脆弱

输出时需要解释为什么推荐某个组合，以及仍然存在哪些风险。

### 8. 最终交付物

第一版应尽量产出：

- 数据体检摘要
- 变量映射表
- 模型设计摘要
- 推荐模型组合解释
- 可复现 Stata `.do` 文件
- Word 回归表
- 简洁结果解释
- 实证分析写作素材
- 文件清单和复现说明

交付物必须强调可复现。

### 9. 质量审查

流程中应包含质量审查：

- 标记不支持的因果表述
- 标记疑似过度挑选结果
- 标记样本量不足
- 标记缺失值过多
- 标记变量定义不清
- 标记需要导师确认的模型选择
- 区分统计显著性和经济显著性

这一点很重要。项目不能被理解成“制造显著性”的工具，而应定位为可复现实证分析流程。

### 10. 打包交付

最终输出应方便用户检查：

- 源代码
- 输出表格
- 摘要说明
- 假设和限制
- 下一步建议

## Skill 架构建议

> 当前实现已调整：第一版先不抽 `skills/_shared/`。输入输出契约、支持方法、工作流契约、实证 section schema 和伦理边界先放在 `skills/starlane-regression/references/` 内。下面保留为早期架构草案背景。

架构应采用开源 skill suite 的组织方式。

第一版不需要一个强制总控 skill。更合适的方式是：

- 仓库 README 作为面向人的全局导航
- 当出现多个稳定 skill 后，再考虑用 `_shared` 保存共享规范和边界
- 每个 skill 都能独立使用
- 每个 skill 内部根据用户所处阶段做局部路由

这样更模块化，也更适合开源协作。

## 第一版正式 Skill

### `starlane-regression`

用途：

从数据和变量映射出发，执行 regression delivery workflow 并产出可复现结果。

适用场景：

- 用户已有数据
- 用户能提供必需变量角色
- 用户希望运行当前支持的回归分析
- 用户希望枚举模型组合并生成最终结果和源码

输入：

- 数据文件路径
- `y`
- `x`
- `cv`
- `panelvar`
- `timevar`
- 中介、调节、异质性、稳健性、工具变量等可选信息
- 预期系数方向
- 执行后端：Python 或 Stata

输出：

- `combination_summary.csv`
- 用户选择的候选行
- 最终结果表
- 可复现源码：`.py` 或 `.do`
- 后端运行说明或日志
- 结果选择说明
- 复现说明
- 局限性说明

边界：

这是第一版唯一正式 skill。它应严格要求必填输入，并透明报告失败原因。除非识别策略充分，否则不能声称结果证明因果关系。

## 未来能力方向

以下能力暂不拆成正式 skill。等 `starlane-regression` 稳定后再决定是否拆分。

### 研究设计能力

未来可能帮助用户把粗略的实证研究想法整理成可执行的研究设计。

可能输入：

- 研究主题
- 已有数据或变量列表
- 目标论文类型
- 已知理论或假设
- 课程、学校或导师限制

可能输出：

- 研究问题
- 假设结构
- 变量映射建议
- 基准模型
- 机制、调节、异质性方案
- 稳健性检验方案
- 可交给 `starlane-regression` 的变量表

边界：

除非识别策略充分，否则不能声称设计能够证明因果关系。

### 结果解释能力

未来可能把实证输出转化为清晰解释和可写入论文的分析内容。

可能输入：

- 回归表
- 变量定义
- 研究假设
- 模型设定
- 推荐结果说明

可能输出：

- 通俗结果解释
- 显著性和方向解释
- 经济含义解释
- 实证分析段落
- 局限和注意事项
- 后续检验建议

边界：

不能过度解释结果。必须区分相关关系、关联关系和因果关系。

## 共享层草案

当前第一版暂不包含 `skills/_shared/`。只有在出现第二个稳定 skill 且多个 skill 真实复用同一套变量规范或边界规则时，再抽共享层。

建议共享文件：

- `user-data-spec.md`：统一变量角色和输入规范
- `supported-methods.md`：当前支持和不支持的方法
- `regression-workflow-contract.md`：回归交付工作流契约
- `ethics-and-boundaries.md`：学术诚信、可复现和因果表述边界

共享层的价值是避免多个 skill 重复写同一套变量规范和边界规则；在只有一个 skill 时，过早抽共享会降低可读性。

## 仓库结构草案

第一版建议结构：

```text
starlane-skills/
├── README.md
├── docs/
│   └── research/
│       └── empirical-analysis-skill-architecture.md
├── scripts/
│   └── stata-code-examples/
└── skills/
    └── starlane-regression/
        ├── SKILL.md
        ├── scripts/
        └── references/
```

第一版正式 skill 只保留 `starlane-regression`。未来的 design、data、writing 能力等第三阶段稳定后再拆。

## 第一里程碑

第一里程碑不是一次性实现所有 skills。

第一里程碑应产出：

- 清晰的共享变量规范
- 能跑通一个真实数据集的 runner 工作流
- 结果解释模板
- 一个样例案例，展示输入、执行、输出和限制

真实案例跑通后，再扩展更多 skills。
