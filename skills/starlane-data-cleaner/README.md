# `starlane-data-cleaner` 的设计说明

## 这个 skill 解决什么问题

实证数据清洗的难点通常不只是把列名改对或把两个文件合并起来，而是先说清楚最终分析数据集应当代表什么：一行对应哪个观察单位，什么字段构成 key，哪些变量必须存在，多个来源之间预期是什么关系，以及多大的样本损失或缺失能够被接受。

如果这些期望没有被明确，清洗很容易退化成对数据“修到能跑”为止：重复值被随手删除、未匹配记录被静默丢弃、缺失被机械填补，最后虽然得到一个文件，却没有证据说明它仍然对应原本的研究对象。这个 skill 的目标是把数据目标、清洗选择和诊断结果组织为同一条可追溯链路。

它面向经管实证中常见的整理、转换、拼接与合并任务，并把重点放在构建可用于分析的数据集，而不是提供一个可任意编程的数据处理环境。

## 核心思路

这个 skill 先建立目标数据集的明确期望，再通过结构化的 `cleaning_plan` 表达清洗和合并操作。plan 同时承载输入、目标、操作和验证条件：它不是一次运行的临时配置，而是解释“为什么得到这份数据”的可复核记录。

实际的数据变换交给稳定的执行器完成，随后用诊断检查结果是否仍符合原先的期望。诊断关注的不是单一的“清洗成功”信号，而是 key 是否唯一、合并双方是否按预期匹配、每一步发生了多少行流失、关键变量是否缺失，以及输出是否留下足够的复现证据。

当问题属于已支持的操作范围时，普通迭代应修改 plan 后重跑同一个执行器。这样，修正是可记录和可复现的。当问题涉及观察单位、匹配规则、变量定义或研究目的等判断时，系统不应把它伪装成技术细节自动处理，而应把证据、可行选项和取舍交还给用户确认。

## 关键设计取舍

### 先定义目标数据集，再开始变换

把观察单位、key、必需变量和预期合并关系放在前面，会增加一点前期沟通成本，但能阻止“以输出文件存在为成功”的错觉。尤其在多表合并中，较高的匹配率、更少的缺失或更多的行数都不天然更好；它们必须与研究对象和原始数据的关系相符。

### 用参数化 plan，而不是一次性脚本

一次性脚本对一个具体数据文件很灵活，却容易把关键选择埋在临时生成的代码里，也容易在下一轮修改中失去前后版本的关系。结构化 `cleaning_plan` 限制了可执行操作的形状，代价是不能立即满足所有特殊需求；换来的则是每次转换、过滤、去重和合并都可以被解释、审阅和重跑。

这个选择并不否认复杂数据任务需要定制逻辑。它只是把“普通清洗的计划修订”和“扩展执行器能力的开发工作”分开，避免为了处理一个个案而悄悄改变所有后续任务的执行语义。

### 把诊断视为交付的一部分

数据集本身不足以说明处理是否可信。这个 skill 因此把行数变化、key、合并结果、缺失和可复现证据与分析数据一起看待。诊断不是为了追求漂亮的指标，而是为了回答：这次处理是否仍满足最初声明的目标；若不满足，偏离发生在哪里，是否可以被研究目的解释。

### 保守地拒绝自动修复

模糊匹配、自动填补、对具有经济含义的变量自动缩尾、随意删除不匹配或重复记录，看似能快速提高匹配率或减少缺失，却常常隐含研究定义和样本选择判断。首版宁愿要求明确规则或用户确认，也不把这些选择包装为无害的默认动作。

## 边界与非目标

这个 skill 不根据表面数据质量指标自行决定什么是“更好”的数据。更高的行数、更低的缺失或更高的匹配率，只有在符合目标观察单位、key、来源关系和研究目的时才有意义。大规模样本流失不必然错误，但不能没有记录和解释。

它不应通过手工修改输出数据来绕开 plan 和诊断，也不应在普通任务中临时改写执行器。否则，交付的数据与其清洗记录会失去可追溯关系，下一次运行也无法可靠复现。

它同样不追求成为任意数据工程或统计修复框架。对于需要外部 crosswalk、复杂模糊匹配、任意自定义脚本或具有实质研究判断的处理，正确的问题不是“能否偷偷自动做掉”，而是应否在明确规则、证据和支持范围后，被作为新的能力正式设计。
