# `starlane-regression` 的设计说明

## 这个 skill 解决什么问题

经管实证回归的困难通常不在于调用某个估计函数，而在于把尚未澄清的研究想法转化为一组可讨论、可复核的设定：研究的对象和变量分别是什么，哪些控制变量有理论依据，标准误应如何处理，哪些扩展检验适用，以及结果可以被解释到什么程度。

如果这些判断只存在于对话里，或由 Agent 临时拼接成一次性脚本，后续很难看出结果建立在哪些选择之上，也很难在用户改变一个研究判断后有序地重做分析。这个 skill 的目标是把这些判断组织为一个可追溯的研究计划，再在该计划基础上完成候选设定比较和最终交付。

它处理的是一条有明确支持范围的回归交付流程，而不是试图覆盖所有计量识别问题。这样的收缩是有意的：比起让 Agent 对任何方法都给出貌似完整的回答，更重要的是让已支持的方法有清楚的输入、选择过程、结果边界和复现材料。

## 核心思路

这个 skill 把研究层与执行层分开。研究层使用 `analysis_plan` 表达用户确认的研究设定；执行层把这些设定转换为具体环境能够运行的参数和源码。前者使用被解释变量、核心解释变量、控制变量、机制或异质性等研究语言，后者才关心估计器和具体实现。这样，用户不必从一开始面对后端参数，Agent 也不应把研究判断藏进脚本细节。

在引导式场景中，计划不是对话结束后补写的一份总结，而是随着基准回归、稳健性、机制、调节、异质性和工具变量等模块逐步确认的同一份结构化草案。它的作用不是替代研究者判断，而是让判断有地方可被检查、修改和复用。

候选设定的汇总阶段把一组共享控制变量和标准误选择下的结果放在同一个可比较单元中。评分可以帮助发现值得检查的候选行，却不能自动决定最终模型。最终选择仍需考虑理论合理性、变量定义、样本和缺失情况、估计稳定性，以及不同检验之间是否形成一致但不过度的证据。

Python 和 Stata 被视为两种执行环境，而不是两套面向用户的分析方法。它们应贯彻同一个研究计划和相同的工作流语义，但可以保留各自生态中合理的估计、输出和源码实现差异。可复现性在这里指的是明确披露选择，并保留能够重现该结果的源码和证据；它不等于要求两个软件逐系数一致。

## 关键设计取舍

### 用一个结构化计划承载研究判断

选择 `analysis_plan`，是为了避免两种常见漂移：先分模块聊天，再另写一份自然语言计划；或者让自然语言计划与实际执行参数各自演化。前者容易遗漏确认过的约束，后者会让用户看到的研究设计和机器实际运行的设定发生分叉。

这并不意味着结构化计划可以替代研究设计。它只能记录已作出的选择，并暴露仍然缺失或含混的部分。变量定义不清、识别假设不成立或理论关系薄弱时，plan 的价值在于使问题可见，而不是为问题提供形式上的通过。

### 枚举候选设定，但不把搜索当作结论

控制变量和标准误等选择常常同时影响估计结果。把候选设定集中呈现，能让这些影响被比较，而不是让一次偶然的脚本输出被默认为唯一答案。与此同时，任何评分或排序都会带来“为显著性寻找规格”的风险，因此候选表被设计为决策辅助材料，而非自动选模器。

这个取舍牺牲了“一键给出最佳模型”的简洁感，换取研究判断的可见性。一个高分候选行仍可能因为控制不足、理论上不合理、样本损失过大或结果脆弱而不适合作为最终设定。

### 统一工作流，不强求统一实现

Python 与 Stata 共享研究层的语义，是为了避免用户在切换工具时重新学习另一套问题表达；允许实现差异，则承认统计软件、估计器和输出生态的真实边界。若为了逐项数值对齐而扭曲任一环境的合理实现，得到的往往是表面一致、维护成本更高的系统。

### 把方法边界写得比能力想象更窄

支持的模型模块覆盖常见的经管回归交付需要，但并不等同于对因果识别的普遍承诺。DID、RDD、PSM、SCM、DML、事件研究等方法被排除，是因为它们各自需要不同的识别诊断、数据结构和解释纪律。把它们塞入现有流程，会把原本清楚的回归工作流变成无法认真维护的万能入口。

## 边界与非目标

这个 skill 不替用户发明变量、数据来源、理论机制或识别策略，也不把统计相关自动解释为因果关系。研究计划和候选比较可以帮助整理已有判断，不能替代对研究设计本身的论证。

它也不是显著性生产工具。候选枚举、评分和结果排序的存在，是为了让不同设定的后果可见；它们不能被用来隐藏不显著结果、挑选缺乏理论依据的规格，或承诺论文通过、发表和结论正确。

最后，它不以覆盖所有实证方法为目标。新增方法意味着新增一套需要被明确表达、运行、诊断和解释的契约，而不只是多加一段回归代码。在没有稳定支持边界前，保持拒绝比给出不可靠的近似支持更符合这个 skill 的目的。
